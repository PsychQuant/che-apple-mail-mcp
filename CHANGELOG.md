## [Unreleased]

### Security
- **`set_background_color` now whitelists the color enum before AppleScript interpolation, closing a pre-existing injection window** ([#116](https://github.com/PsychQuant/che-apple-mail-mcp/issues/116) ŌĆö surfaced during [#104](https://github.com/PsychQuant/che-apple-mail-mcp/issues/104) PR-A verify by Codex P2 / DA-F / Logic LOW#1). The `color` argument flowed from the MCP client through `Server.swift`'s `set_background_color` handler into `MutationToolScriptBuilder.buildSetBackgroundColorScript`'s `set background color of <ref> to \(color)` line **without any validation** ŌĆö a malicious or buggy caller passing newline-bearing (`"red\n        do shell script \"ŌĆ”\""`), quote-bearing, or semicolon-chained payloads could inject arbitrary AppleScript that ships into `osascript`. A whole-codebase sibling sweep confirmed this is the **only** externally-supplied enum-shaped `String` slot raw-interpolated across all 6 AppleScript builder files (other slots are typed `Bool`/`Int` or pre-escaped refs). Defense-in-depth fix: (1) new `backgroundColorWhitelist: Set<String>` constant co-located with the builder ŌĆö single source of truth, lowercase-strict, exact-match against Apple Mail's 8-value enum; (2) `Server.swift` handler `guard`s membership before delegating, throws `MailError.invalidParameter` with the valid set echoed back to the caller; (3) `buildSetBackgroundColorScript` adds a `precondition` for programmer-error catch if any internal caller bypasses the handler gate (crash beats silent injection). 5 new tests pin the constant against drift, exercise injection-payload rejection (newline / quote / semicolon), case-strictness (`"Blue"` / `"BLUE"` rejected), and whitespace-strictness (`"blue "` / `""` rejected). Existing 2 happy-path tests (UUID path + display_name fallback for `"blue"` / `"green"`) stay green. **Backward compat preserved**: callers using the 8 documented enum values (blue, gray, green, none, orange, purple, red, yellow) see identical behavior.

## [2.9.0] - 2026-05-18

### Fixed
- **`list_attachments` no longer silently returns `[]` for Outlook 16 Windows attachments with RFC 2231 + nested RFC 2047 filenames** ([#99](https://github.com/PsychQuant/che-apple-mail-mcp/issues/99)). Outlook 16 Windows clients double-encode non-ASCII filenames by embedding an RFC 2047 encoded-word (`=?charset?B?...?=`) inside the RFC 2231/5987 percent-encoded payload ŌĆö common when sending Chinese filenames Ōēź 64 UTF-8 bytes (forces RFC 2231 continuation split). Pre-fix: `MIMEParser.resolveFilename` correctly decoded the outer RFC 2231 layer (assembled `filename*0*` + `filename*1*` continuation, percent-decoded with charset) but returned the inner encoded-word verbatim (`=?utf-8?B?5Lit...?=\t=?utf-8?B?6KGo...?=`). `crossValidateAttachments` then compared the parser's raw encoded-word against Mail.app's SQLite-cached human-readable name ŌåÆ no intersection ŌåÆ all rows dropped ŌåÆ `[]`. Symptom: `list_attachments` returns empty even though Mail.app UI shows downloadable attachments; subsequent `save_attachment` raises misleading `-1728 "Can't get account"` (Tier 1 throws `attachmentNotFound` because user-supplied filename doesn't match the parser's raw encoded-word, falls through to AppleScript). Post-fix: new `decodeRFC2047IfApplicable` helper applies an idempotent RFC 2047 second pass after the outer decode, gated on a strict full-string regex match for encoded-word grammar (`^=\?[^?\s]+\?[BQbq]\?[^?]*\?=(\s*=\?[^?\s]+\?[BQbq]\?[^?]*\?=)*$`) to prevent false-positive corruption of filenames literally containing `=?...?=` substrings. Applied at all 4 return points in `resolveFilename` (RFC 5987 path, RFC 2231 continuation path, plain `filename=` path, legacy Content-Type `name=` path). Affected: any sender using Outlook 16 / Windows + Chinese filename + multi-byte UTF-8 (academic / government / corporate forms commonly trigger this). Companion `save_attachment` `-1728` symptom resolves as side effect of Tier 1 now succeeding (no AppleScript fallthrough needed); the underlying display_name collision defect itself is independently fixed by [#101](https://github.com/PsychQuant/che-apple-mail-mcp/issues/101). 4 new tests: exact Outlook 16 fixture from issue body (decodes to `õĖŁÕż«ńĀöń®ČķÖóµ¢░ķĆ▓ĶüśÕā▒õ║║ÕōĪķ½öµĀ╝µ¬óµ¤źĶĪ©-1110331ĶŻĮĶĪ©.pdf`), RFC 5987 single-segment with nested encoded-word variant, plain filename regression, partial-substring defense.
- **5 single-message mutation tools (`mark_read`, `flag_email`, `set_flag_color`, `set_background_color`, `mark_as_junk`) now accept `account_id` for multi-account-same-display_name disambiguation** ([#104](https://github.com/PsychQuant/che-apple-mail-mcp/issues/104) PR-A). Same `-1728 / -1719` failure mode as [#101](https://github.com/PsychQuant/che-apple-mail-mcp/issues/101)'s `save_attachment` ŌĆö `msgRef`'s `account "<display_name>"` AppleScript selector non-deterministically picks the wrong account when two accounts share a display_name (iCloud catch-all alias + Gmail same address). Pre-fix: any of the 5 mutation tools could fail or operate on the wrong account's message. Post-fix: each tool accepts an optional `account_id` (Mail.app's globally-unique UUID); when provided, AppleScript uses `(account id "<UUID>")` selector instead. Implementation reuses Phase 0's shared `AppleScriptRefBuilder.resolveMsgRef(...)` chokepoint ŌĆö single resolver feeds all 5 tools (and `save_attachment`), so PR-B (move/copy/delete), PR-C (compose), PR-D (mailbox CRUD) become pure call-site swaps. **Backward compat preserved**: omitting `account_id` falls back to the legacy display_name path. 19 new tests (9 resolver + 10 builder, 2 per mutation tool: UUID path + display_name fallback).
- **`save_attachment` no longer fails with `-1728 / -1719` when two accounts share a `display_name`** ([#101](https://github.com/PsychQuant/che-apple-mail-mcp/issues/101)). Mail.app's AppleScript `account "<display_name>"` selector is non-unique under multi-account-same-display_name configurations (common: iCloud catch-all alias + Gmail with the same address). Pre-fix: any `save_attachment` call that falls through Tier 1 (SQLite + `.emlx` fast path) to Tier 2 (AppleScript fallback) hits `mailboxRef`'s `account "<display_name>"` selector ŌĆö which non-deterministically picks the wrong account ŌåÆ either `-1728 "Can't get account"` or `-1719 "Invalid mailbox index"`. The `account_id` is also now plumbed through `SearchResult` and **exposed in `search_emails` JSON output** (populated from the SQLite `mailboxes` URL via `MailboxURL.decode`), so callers can pass it through directly from `search_emails` results to `save_attachment` ŌĆö completing the documented discovery ŌåÆ disambiguation workflow. AppleScript `(account id "...")` syntax empirically verified on macOS 14 ŌĆö error modes (`-1719 / -1728`) match the display_name path, so existing error handling continues to apply. New `SaveAttachmentScriptBuilder` free-function module (matching `ComposeScriptBuilder` pattern) makes the AppleScript generation testable without spinning up the actor. 15 new tests added (3 structural `SearchResult.accountId` + 7 builder tests + 5 `search_emails` JSON formatter tests pinning the disambiguation discovery contract). Scope strictly limited to `save_attachment`; the ~14 other AppleScript-routed tools (`get_email`, `mark_read`, etc.) are tracked at [#104](https://github.com/PsychQuant/che-apple-mail-mcp/issues/104) for follow-up sweep.

### Fixed
- **`list_emails` SQLite fast path now has do/catch fallback + AppleScript path optimized 3├Ś IPC reduction** ([#89](https://github.com/PsychQuant/che-apple-mail-mcp/issues/89)). The bug report (filed against v2.1.0 pre-SQLite) documented 14+ minute response times on a 92k-message Gmail INBOX. v2.2.0+ already shipped a SQLite fast path that's O(LIMIT) regardless of mailbox size, fixing the core issue. This PR adds two defense-in-depth improvements: (1) Server.swift `case "list_emails"` now wraps the SQLite call in do/catch with stderr log + AppleScript fallthrough (mirrors the established pattern from save_attachment #12, get_email_metadata #71, list_attachments #24); pre-fix, a SQLite throw would propagate to the caller instead of falling back. (2) MailController.listEmails (AppleScript fallback path, rarely hit) batched the previously-separate `subjects` + `senders` + `ids` scripts into ONE ŌĆö single IPC, single mailbox resolution. Also dropped the unnecessary `count of messages of mb` guard (which was the actual 14-min bottleneck on huge IMAP mailboxes ŌĆö `messages 1 thru N` clamps internally, no need to pre-count). Net: 3├Ś IPC reduction + count-of-messages bottleneck removed when SQLite is unavailable.
- **Malformed multipart now surfaces as parse-failure instead of silently filtering all attachments** ([#26](https://github.com/PsychQuant/che-apple-mail-mcp/issues/26)). Pre-fix, `MIMEParser.enumerateAttachmentNames` returned `Set()` when the top-level Content-Type declared `multipart/*` but the body was unparseable (missing boundary parameter, all children failing header/body split, depth limit). The `list_attachments` cross-validation filter then used that empty set to drop ALL SQLite-cached attachment rows ŌĆö net effect: users saw 0 attachments on legitimately-broken `.emlx` files even though Mail.app's index still had the names. Fix: throw `MailSQLiteError.emlxParseFailed` when top-level multipart cannot be split at all (depth==0 with no child parsed). Server.swift handlers already wrap `attachmentNames` in `do/catch` and fall back to unvalidated SQLite metadata (per #28's `crossValidateAttachments` helper), so users now see the SQLite-cached names instead of the empty post-filter result. Non-multipart inputs (single-part `text/plain`) and valid multipart with legitimately zero attachments (all children are text/* with no filename) continue to return empty set without throwing ŌĆö pre-#26 behavior preserved for those cases.

### Fixed

The 100 MB size guard is applied to the externalised bytes too. Architecturally: the SQLite + .emlx fast path now handles `.partial.emlx` end-to-end; the AppleScript fallback at `Server.swift:1017` is no longer reached for this common pattern.

## [2.2.0] - 2026-04-14

### Performance
- **`save_attachment` is now 10ŌĆō100├Ś faster** ([#12](https://github.com/PsychQuant/che-apple-mail-mcp/issues/12)). The previous implementation went through AppleScript IPC (`tell Mail.app to save att in POSIX file ...`), taking 1ŌĆō3 seconds per attachment. The new fast path reads the `.emlx` file directly, parses the MIME multipart structure, decodes the matching part, and writes it ŌĆö all in-memory ŌĆö for sub-50 ms latency. Bulk operations (e.g., `/archive-mail` attachment backfill) drop from ~30 s to under 1 s for 14 attachments.

### Added
- **`MIMEParser.parseAllParts(_:headers:) -> [MIMEPart]`** ŌĆö a non-lossy MIME enumeration API that walks the multipart tree and returns every part (text, html, attachments, inline images, nested multipart). Complements (does not replace) the existing `parseBody` text-extraction API. The new API is hardened against multipart bombs with a depth limit of 8.
- **`MIMEPart` struct** in `Sources/MailSQLite/MIMEPart.swift` ŌĆö a `Sendable`, `Equatable` value type that carries `headers`, `contentType`, `contentTypeParams`, `contentDisposition`, `filename`, `rawBytes`, and eagerly-decoded `decodedData` for each part. Filename resolution honors RFC 2231 / RFC 5987 continuation encoding and percent-decoded UTF-8 (so Exchange / Gmail CJK filenames work).
- **`EmlxParser.saveAttachment(rowId:mailboxURL:attachmentName:destination:)`** in `Sources/MailSQLite/AttachmentExtractor.swift` ŌĆö direct-from-filesystem attachment extraction. Resolves `.emlx` path via the existing `mailStoragePathOverride` test hook (#9), strips the Apple wrapper, walks all MIME parts, finds first match by filename (first-match semantics), and writes `decodedData.write(to: destination, options: .atomic)`.
- **Typed errors**: `MailSQLiteError.attachmentNotFound(name:)` and `MailSQLiteError.attachmentTooLarge(name:size:limit:)` so the dispatcher can distinguish fallback triggers.
- **26 new tests** (156 total now, was 130): 6 for `MIMEPart`, 13 for `MIMEParser.parseAllParts` + filename decoding, 7 for `AttachmentExtractor` end-to-end with fixture `.emlx` files.
- **4 fixture files** in `Tests/MailSQLiteTests/Fixtures/`: ASCII attachment, CJK filename (RFC 5987), nested multipart, duplicate filename. Plus expected-payload `.bin` files for byte-level assertion.

### Changed
- **`save_attachment` MCP tool dispatcher** (`Server.swift:939`) now uses a **two-tier catch** pattern matching `get_email`'s precedent: SQLite + `.emlx` fast path runs in its own `do/catch`; any thrown error falls through to the legacy `MailController.saveAttachment` AppleScript call. The two `do/catch` blocks are intentionally **not** collapsed into one ŌĆö that mistake caused `#9`'s `get_emails_batch` regression and we explicitly avoid it here. The fallback path logs the cause to stderr (`SQLite save_attachment fast path failed: ..., falling through to AppleScript`) so silent fallbacks are observable in production.
- **`MIMEParser.parseBody` is unchanged** ŌĆö preserved as the hot path used by `get_email` / `list_emails`. The two parallel APIs are cross-verified by `testParseAllPartsAndParseBodyAgreeOnTextBody`.

### Non-Goals (deferred)
- **Streaming / `FileHandle` pipeline**: out of scope. The fast path is in-memory only. Attachments larger than 100 MB throw `attachmentTooLarge`, which the dispatcher catches and falls through to the AppleScript path (Mail.app handles streaming write internally). 99% of real attachments are well under this limit.
- **Inline image `cid:` resolution**: `save_attachment` writes part bytes, not HTML rewrites. Inline images with filenames are still saveable, but `<img src="cid:...">` references in HTML body are unchanged.
- **Exotic transfer encodings** (`uuencode`, `binhex`): unsupported on the fast path; falls through to AppleScript.
- **GitHub Actions automation** for releases: still tracked under [#13](https://github.com/PsychQuant/che-apple-mail-mcp/issues/13). For now, releases are published via `./scripts/release.sh`.

### Spec
- **`emlx-parser` capability** gains a new `Attachment extraction from emlx` requirement with 6 scenarios (ASCII PDF, CJK filename, fallback on throw, first-match duplicate, parseAllParts Ōćä parseBody consistency, large-attachment size-based fallback). See `openspec/changes/save-attachment-fast-path/specs/emlx-parser/spec.md`.

## [2.1.1] - 2026-04-14
